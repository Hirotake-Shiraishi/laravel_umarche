# Laravel 次ステップ学習アプリ提案

## いま身についていること（このプロジェクトで）

- **マルチロール認証**（User / Owner / Admin、Breeze）
- **CRUD・リレーション**（Shop → Product → Stock、カテゴリ、画像）
- **外部連携**（Stripe 決済、Intervention Image）
- **非同期処理**（キュー、ジョブ、メール送信）
- **Blade** による画面構築

一方で、**注文テーブル・注文履歴**は未実装で、**API** はほぼ未使用です。

---

## 実務でよく使う「次の学習候補」3つ

実務では「API＋フロント分離」「予約・スケジュール」「SaaS・サブスク」がよく出てきます。いずれも今のECで触った要素を活かしつつ、新しいパターンを学べます。

---

### 候補1: **API + フロント分離アプリ（最優先でおすすめ）**

**作るもの例**: 既存ECの「管理画面」を API 化し、Vue / React / Inertia で作り直す、または「タスク管理・簡易CRM」を API + SPA で作る。

**学べる実務スキル**

- **Laravel Sanctum**（SPA / モバイル用トークン認証）
- **RESTful API** 設計（リソース、HTTPメソッド、ステータスコード）
- **API Resource**（`php artisan make:resource`）でJSONの形を統一
- **CORS・認証ミドルウェア** の扱い
- フロント（Vue/React/Inertia）との役割分担

**このプロジェクトとの接続**

- 既存の `routes/api.php` はほぼ未使用なので、商品・カート・オーナー用APIを追加する形で「API版EC」として拡張する学習も可能。
- 認証は `auth:api` を Sanctum に移行する流れで学べる。

**実務での重要度**: 非常に高い（モバイルアプリ・SPA・他システム連携の基盤）

---

### 候補2: **予約・スケジュール系アプリ**

**作るもの例**: 美容院・教室・会議室の「予約システム」。予約枠（日時・リソース）の管理、重複防止、キャンセル・リマインダーメール。

**学べる実務スキル**

- **日付・時刻の扱い**（Carbon、タイムゾーン、営業時間）
- **スケジュール・枠管理**（1対多の「枠」と「予約」、在庫的な考え方の応用）
- **バリデーション**（「その枠は既に予約済み」などの業務ルール）
- **通知**（予約確定・リマインダー・キャンセルメール、キュー活用の復習）
- 必要なら **カレンダーUI**（フロント or ライブラリ）との連携

**このプロジェクトとの接続**

- 在庫（t_stocks）の「数量」の代わりに「予約枠」を管理するイメージで、考え方が流用できる。
- オーナー／管理者の権限分けは、今の Owner/Admin の延長で設計できる。

**実務での重要度**: 高い（予約・施設利用・採用面接などで頻出）

---

### 候補3: **SaaS・サブスクリプション系**

**作るもの例**: 「月額プランで使うツール」の最小構成。プラン（無料/有料）・課金周期・解約と、Stripe Billing または Paddle などでの継続課金。

**学べる実務スキル**

- **プラン・課金周期**のテーブル設計（plans, subscriptions）
- **Stripe Billing**（サブスク、顧客・サブスク作成、Webhook）
- **Webhook** の受信・検証・冪等性（解約・更新・失敗の反映）
- **中間テーブル**（users ↔ plans、有効期限・キャンセル日）
- **権限・機能制限**（プランごとの「使える機能」の分岐）

**このプロジェクトとの接続**

- すでに Stripe 決済を入れているので、**Stripe Billing** に広げる形で「サブスク」を追加する流れが自然。
- Owner を「テナント」とみなし、Owner ごとにプラン（無料/有料）を紐づける設計にすると、マルチテナントの理解が深まる。

**実務での重要度**: 高い（SaaS・会員制サービスでは必須）

## 選び方の目安


| やりたいこと                                  | おすすめ                  |
| --------------------------------------- | --------------------- |
| フロントエンドやモバイルと組み合わせたい、転職で「API開発」をアピールしたい | **候補1（API + フロント分離）** |
| 予約・施設・採用など「枠管理」の仕事に活かしたい                | **候補2（予約・スケジュール）**    |
| 月額課金・SaaS の仕組みを理解したい                    | **候補3（SaaS・サブスク）**    |


**まず1つ選ぶなら**: 実務の汎用性と「今のプロジェクトの延長で学べる」ことを考えると、**候補1（API + フロント分離）** を最優先に進め、その次に候補2か3のどちらかを選ぶのがおすすめです。

---

## 補足: 今のECサイトの「足し算」でできること

- **注文テーブル・注文履歴**: 現プロジェクトに `orders` / `order_items` を追加し、Stripe 成功時に保存・一覧表示するだけでも、実務に近い「注文管理」の練習になる。
- **API 化**: 同じモデル・ロジックを使い、`api.php` と API Resource で JSON を返す版を並行して作ると、候補1の入門として最適。

この2つは「次の新規アプリ」と並行して、今のリポジトリで小さく進めてもよいです。

---

## まとめ

- **次に作るアプリ**としては、**API + フロント分離（候補1）** を最優先にし、**予約系（候補2）** か **SaaS・サブスク（候補3）** のどちらかを興味・目標に合わせて選ぶのがよいです。
- あわせて、今のECに **注文テーブル・注文履歴** と **API の足し算** をすると、実務でよく使う「CRUD + 決済 + API」の流れを一通り経験できます。
